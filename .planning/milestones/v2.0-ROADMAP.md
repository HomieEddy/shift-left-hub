# Milestone v2.0: Workspace Platform

**Status:** ✅ SHIPPED 2026-06-14
**Phases:** 9-16
**Total Plans:** 34 plans

## Overview

Transform the app from an IT-specific helpdesk into a general-purpose, plug-and-play knowledge platform that any team can adopt. Multi-tenant workspaces, document ingestion, BYO LLM, domain-agnostic assistant.

## Phases

### Phase 9: Workspace Foundation

**Goal**: Multi-tenant workspace isolation — users can create workspaces, all domain data is scoped by workspace_id, and existing v1.0 data is migrated to a default workspace.
**Depends on**: Nothing (v1.0 shipped)
**Plans**: 4 plans

Plans:

- [x] 09-01: Database schema + JPA entities (Flyway V3, Workspace, WorkspaceAwareEntity)
- [x] 09-02: Security + AOP + Vector filtering (JWT claims, ContextHolder, FilterAspect)
- [x] 09-03: Migration + Admin API (Default Workspace seeder, AdminWorkspaceController)
- [x] 09-04: Admin workspace frontend (list, create, assign users)

**Details:**
- Workspace, WorkspaceMember entities with Flyway V3 migration
- Hibernate @Filter + AOP for workspace-scoped data isolation
- JWT workspace_id claim for authentication/authorization
- pgvector metadata filtering for workspace-scoped similarity searches
- Default Workspace migration for v1.0 data compatibility

### Phase 10: Document Ingestion + BYO LLM

**Goal**: Users can upload documents for async Spring AI ETL processing and configure their own OpenAI-compatible LLM endpoint per workspace.
**Depends on**: Phase 9
**Plans**: 4 plans

Plans:

- [x] 10-01: Database schema + JPA entities (Flyway V4, Document, DocumentChunk, WorkspaceLlmConfig)
- [x] 10-02: Document upload + async ETL pipeline (event-driven)
- [x] 10-03: Workspace LLM config + ChatModelRegistry
- [x] 10-04: Frontend admin document management + LLM config UI

**Details:**
- Drag-and-drop upload for markdown, text, PDF files
- Spring AI ETL pipeline: UPLOADED → PARSING → CHUNKING → EMBEDDING → READY/FAILED
- SHA-256 content hash deduplication
- WorkspaceLlmConfig with encrypted API keys (Spring TextEncryptor)
- WorkspaceChatModelRegistry for cached, workspace-scoped LLM routing

### Phase 11: Domain-Agnostic AI

**Goal**: The AI assistant becomes domain-agnostic — workspaces define their own taxonomy, customize system prompts, and search across both articles and document chunks.
**Depends on**: Phase 10
**Plans**: 6 plans

Plans:

- [x] 11-01: Database schema + JPA entities (Flyway V5, Category, system_prompt column, doc chunk tsvector)
- [x] 11-02: Category admin API + service
- [x] 11-03: System prompt + AI customization (template variables)
- [x] 11-04: Unified hybrid search (document chunk FTS + vector, three-way RRF merge)
- [x] 11-05: Taxonomy management UI (tree view, category badges)
- [x] 11-06: AI customization UI + search display

**Details:**
- Category/subcategory taxonomy per workspace
- System prompt template variables: {workspace_name}, {domain}, {categories}
- Unified RRF-ranked hybrid search across articles and document chunks
- Category badges and source citations in search results and chat

## Milestone Summary

**Key Decisions:**
- Row-level workspace_id isolation (not schema-per-tenant) — preserves Flyway, Testcontainers, pgvector
- Hibernate @Filter + AOP for defense-in-depth data isolation
- Combined DOC + LLM in Phase 10 (shared embedding infrastructure)
- Default Workspace migration for v1.0 backward compatibility
- Master + per-workspace seeders with idempotency
- Warm slate/charcoal palette for domain-agnostic branding
- SVG favicon (book icon) replacing favicon.ico

**Issues Resolved:**
- v1.0 data migration to multi-tenant workspaces
- Frontend Angular Style Guide compliance
- Seeding from IT-specific to multi-domain

**Technical Debt Incurred:**
- CR-01: Native SQL queries bypass Hibernate @Filter (cross-workspace data leakage risk)
- CR-02: Vector search filter uses string concatenation
- Phase 9: WR-01–04 (HTTP error codes, missing migration, missing checks)
- Phase 15: WR-01–04 (.docx fallback, null MIME handling, MIME-extension mismatch, error messages)
