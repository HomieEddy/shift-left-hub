  <action>
    Create `.github/workflows/ci.yml`:

    ```yaml
    name: CI
    
    on:
      pull_request:
        branches: [master, main]
      push:
        branches: [master, main]
    
    env:
      SONAR_PROJECT_KEY: shiftleft-knowledge-hub
      SONAR_ORG: shiftleft
      SONAR_HOST_URL: https://sonarcloud.io
    
    jobs:
      backend:
        name: Backend (Checkstyle + SpotBugs + Tests + Coverage)
        runs-on: ubuntu-latest
    
        services:
          postgres:
            image: pgvector/pgvector:0.8.0-pg16
            env:
              POSTGRES_DB: shiftleft
              POSTGRES_USER: shiftleft
              POSTGRES_PASSWORD: shiftleft
            ports:
              - 5432:5432
            options: >-
              --health-cmd pg_isready
              --health-interval 10s
              --health-timeout 5s
              --health-retries 5
    
        steps:
          - uses: actions/checkout@v4
            with:
              fetch-depth: 0  # SonarQube needs full git history
    
          - name: Set up JDK 21 (Temurin)
            uses: actions/setup-java@v4
            with:
              java-version: '21'
              distribution: 'temurin'
              cache: 'maven'
    
          - name: Cache Maven dependencies
            uses: actions/cache@v4
            with:
              path: ~/.m2/repository
              key: ${{ runner.os }}-maven-${{ hashFiles('backend/pom.xml') }}
              restore-keys: |
                ${{ runner.os }}-maven-
    
          - name: Run backend verify (Checkstyle + SpotBugs + Unit + Integration Tests + JaCoCo)
            run: ./mvnw verify -B
            working-directory: backend
            env:
              SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/shiftleft
              SPRING_DATASOURCE_USERNAME: shiftleft
              SPRING_DATASOURCE_PASSWORD: shiftleft
              # Use Testcontainers with the service container's PostgreSQL as the TC host
              DOCKER_HOST: unix:///var/run/docker.sock
              TESTCONTAINERS_HOST_OVERRIDE: postgres
    
          - name: SonarQube Cloud Scan
            if: always()  # Runs even if tests fail (informational per D-02)
            run: >
              ./mvnw sonar:sonar -B
              -Dsonar.projectKey=${{ env.SONAR_PROJECT_KEY }}
              -Dsonar.organization=${{ env.SONAR_ORG }}
              -Dsonar.host.url=${{ env.SONAR_HOST_URL }}
              -Dsonar.token=${{ secrets.SONAR_TOKEN }}
              -Dsonar.coverage.jacoco.xmlReportPaths=target/site/jacoco/jacoco.xml
            working-directory: backend
            continue-on-error: true  # Non-blocking per D-02
    
      frontend:
        name: Frontend (Lint + Test + Build)
        runs-on: ubuntu-latest
    
        steps:
          - uses: actions/checkout@v4
    
          - name: Set up Node.js 22
            uses: actions/setup-node@v4
            with:
              node-version: '22'
    
          - name: Set up pnpm
            uses: pnpm/action-setup@v4
            with:
              version: 10
    
          - name: Cache pnpm store
            uses: actions/cache@v4
            with:
              path: ~/.local/share/pnpm/store
              key: ${{ runner.os }}-pnpm-store-${{ hashFiles('frontend/pnpm-lock.yaml') }}
              restore-keys: |
                ${{ runner.os }}-pnpm-store-
    
          - name: Install dependencies
            run: pnpm install --frozen-lockfile
            working-directory: frontend
    
          - name: Run ESLint
            run: pnpm lint
            working-directory: frontend
    
          - name: Run unit tests with coverage
            run: pnpm test -- --watch=false --code-coverage --browsers=ChromeHeadless
            working-directory: frontend
    
          - name: Build
            run: pnpm build
            working-directory: frontend
    ```

    NOTE: 
    - The `TESTCONTAINERS_HOST_OVERRIDE` env var is needed when Testcontainers tries to start containers inside the CI runner but the PostgreSQL service container is used for integration tests. If Testcontainers attempts to start its own container, it may conflict. The backend CI uses the service container PostgreSQL directly via `SPRING_DATASOURCE_URL`, so integration tests that extend `AbstractIntegrationTest` can either use the service container's PostgreSQL or start their own Testcontainers container. 
    - For simplicity, ensure `AbstractIntegrationTest` uses `@DynamicPropertySource` to point to the service container when `SPRING_DATASOURCE_URL` is already set, falling back to Testcontainers auto-start otherwise. Or simpler: integration tests in CI will use the same PostgreSQL from the service container. The Testcontainers dependency will be present but may not start a new container if the datasource URL is already set.
    
    **Important refinement**: To avoid Testcontainers trying to start a Docker container in the CI runner (which requires Docker-in-Docker or the Docker socket), make the integration tests configurable to connect to the service container PostgreSQL directly when `SPRING_DATASOURCE_URL` is provided. Update `AbstractIntegrationTest` to check if a datasource URL is already set:
    
    ```java
    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        // If CI provides the PostgreSQL URL (from service container), use it
        // Otherwise, start a Testcontainers-managed container
        String ciUrl = System.getProperty("spring.datasource.url");
        if (ciUrl != null && !ciUrl.isEmpty()) {
            // CI mode — service container handles PostgreSQL
            registry.add("spring.datasource.url", () -> ciUrl);
            registry.add("spring.datasource.username", () -> System.getProperty("spring.datasource.username", "shiftleft"));
            registry.add("spring.datasource.password", () -> System.getProperty("spring.datasource.password", "shiftleft"));
        } else {
            // Local mode — Testcontainers starts the container
            registry.add("spring.datasource.url", postgres::getJdbcUrl);
            registry.add("spring.datasource.username", postgres::getUsername);
            registry.add("spring.datasource.password", postgres::getPassword);
        }
    }
    ```
    
    This ensures the same `AbstractIntegrationTest` works in both local dev (auto-start container) and CI (use service container).

    Alternatively, make the approach even simpler: Use Testcontainers in the CI job as well (it works if Docker is available). The `pgvector/pgvector:0.8.0-pg16` service container in CI is available as a PostgreSQL server, but Testcontainers can also start its own container. The safest approach is:
    
    - In CI: Don't start the service container. Just let Testcontainers start its own `pgvector/pgvector:0.8.0-pg16` container via the `@ServiceConnection` mechanism. This requires Docker-in-Docker or Docker socket access. GitHub Actions hosted runners have Docker installed.
    - Remove the service container from the backend CI job and rely on Testcontainers directly.
    
    Go with the Testcontainers-direct approach for the CI job (cleaner — single source of truth):
    ```yaml
    - name: Run backend verify (Checkstyle + SpotBugs + Unit + Integration Tests + JaCoCo)
      run: ./mvnw verify -B
      working-directory: backend
    ```
    No need for service container or env overrides — Testcontainers handles everything. This is the recommended approach for Spring Boot + Testcontainers in GitHub Actions.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI Pipeline ↔ GitHub Actions | The CI workflow runs in GitHub Actions runners. Backend dependencies are resolved from Maven Central, npm packages from npm registry. Docker images pulled from Docker Hub/GitHub Container Registry. |
| CI Pipeline ↔ SonarCloud | Backend CI sends coverage reports to SonarCloud. Only project key and org are in CI YAML — the secret `SONAR_TOKEN` is stored as a GitHub Actions secret. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-07-01 | S | SONAR_TOKEN secret | mitigate | Token is stored as a GitHub Actions secret (`secrets.SONAR_TOKEN`), never exposed in logs. The `sonar:sonar` Maven goal reads it from the secret. |
| T-08-07-02 | S | Docker image supply chain | accept | `pgvector/pgvector:0.8.0-pg16` is a pinned public image. Testcontainers pulls the same image in CI. Image is from a trusted publisher (pgvector org). |
| T-08-07-03 | S | pnpm lockfile integrity | mitigate | CI uses `--frozen-lockfile` to ensure no dependency drift. Any change requires a new lockfile commit. |
| T-08-07-04 | T | CI workflow injection | accept | The workflow checks out PR code and runs `mvnw verify`. The mvnw wrapper script could be modified by the PR. For a solo dev portfolio project, this risk is accepted. Production would use pinned action versions and `pull_request_target` with controlled checkout. |
</threat_model>
