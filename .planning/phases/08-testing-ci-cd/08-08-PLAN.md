<task type="auto">
  <name>Task 2: Create pre-push hook with full verification suite</name>
  <files>.husky/pre-push</files>
  <action>
    Create `.husky/pre-push`:

    ```bash
    #!/bin/sh
    # Pre-push hook — runs full verification suite before allowing push.
    # Blocks push on any failure.
    
    echo "🔍 Running pre-push checks..."
    
    # Backend: Checkstyle + SpotBugs + unit tests + integration tests + JaCoCo
    echo "--- Backend: mvn verify ---"
    cd backend && ./mvnw verify -B
    if [ $? -ne 0 ]; then
      echo "❌ Backend checks failed. Push blocked."
      exit 1
    fi
    cd ..
    
    # Frontend: ESLint
    echo "--- Frontend: ESLint ---"
    cd frontend && pnpm lint
    if [ $? -ne 0 ]; then
      echo "❌ Frontend lint failed. Push blocked."
      exit 1
    fi
    
    # Frontend: Tests
    echo "--- Frontend: Unit Tests ---"
    pnpm test -- --watch=false --browsers=ChromeHeadless
    if [ $? -ne 0 ]; then
      echo "❌ Frontend tests failed. Push blocked."
      exit 1
    fi
    cd ..
    
    echo "✅ All pre-push checks passed!"
    ```

    Make the file executable:
    ```bash
    git update-index --chmod=+x .husky/pre-push
    ```

    On Windows, the `.husky/pre-push` file must have LF line endings (not CRLF) for Husky to work. Ensure the file uses Unix line endings.

    For Windows compatibility, the pre-push hook could use the node-based approach:
    ```bash
    node --eval "
      const { execSync } = require('child_process');
      const chalk = ... 
    "
    ```
    
    But the simpler shell approach with `#!/bin/sh` works if git-bash is available. On Windows with Git for Windows, `#!/bin/sh` maps to the Git Bash shell which has `sh.exe` available.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-08-01 | S | Pre-push hook skipping | mitigate | Husky hooks are committed to the repo and checked out with the code. Users cannot push without running hooks unless they use `--no-verify`. This is opt-in by convention. CI provides the hard gate. |
| T-08-08-02 | S | Pre-commit modifies files unexpectedly | mitigate | Pre-commit runs `prettier --write` which may reformat files. This is intentional and should be part of the developer workflow. ESLint `--fix` only applies auto-fixable rules. Non-auto-fixable errors fail the commit and require manual fixing. |
</threat_model>
